0Quick Start

Two environment variables decide which mode the harness runs in. No config files. No build step — Vite compiles TypeScript on the fly.

Browse every control in a workspace with card layout, search, build badges, and package size tracking.

Bash # Gallery mode — browse all controls
cd PCF-Workbench/harness
PCF_WORKSPACE_ROOT="/path/to/pcf-controls" npx vite --port 8181

Load a single control directly for focused development and testing.

Bash # Single control mode
cd PCF-Workbench/harness
PCF_CONTROL_PATH="/path/to/MyControl/MyControl" npx vite --port 8181

PowerShell syntax for setting environment variables.

PowerShell # Gallery mode (PowerShell)
cd PCF-Workbench\harness
$env:PCF_WORKSPACE_ROOT = "C:\path\to\pcf-controls"
npx vite --port 8181

3Network Conditioning

Simulate real-world network conditions to test offline-first patterns, degraded connectivity, and Mobile Offline Profile behavior.

Online0 ms
OfflineRoutes to offline store
Slow 3G2000 ms
Fast 3G500 ms
CustomUser-defined ms
How it works — mirrors real Dynamics routing

The WebAPI shim mirrors the real Dynamics 365 routing model: context.webAPI auto-routes between online and offline stores, while webAPI.online and webAPI.offline target specific stores directly. In Offline mode, online calls reject and auto-routing falls back to the local store. In throttled modes, an artificial delay is injected via setTimeout. A status badge shows the active network state; custom mode lets you set any latency in milliseconds.

8Performance Monitoring

Real-time metrics to identify rendering bottlenecks, slow WebAPI calls, and memory leaks.

Render Timeline

Last 50 updateView durations displayed as a sparkline. A red budget line at 16 ms marks the 60 fps threshold.

DOM & Heap Metrics

Tracks render count, DOM node count inside the container, and JS heap usage. Captures heap snapshots labeled by render cycle for leak hunting.

WebAPI Call Log

Every WebAPI operation (last 100) with method, entity type, duration, response size, record count, and the OData query string. Surfaces slowest, largest, and failed requests.

9Lifecycle Monitor Health checks

Track every PCF lifecycle method (init, updateView, getOutputs, destroy, notifyOutputChanged) with precise timing and automated health checks.

Full Lifecycle Log

Each hook logged with duration, parameter snapshot, and call order. Spot unexpected re-renders or missing notifyOutputChanged callbacks.

Automated Health Checks

Flags common anti-patterns: updateView loops, excessive re-renders from unstable callbacks, missing destroy cleanup, render durations over budget.

Virtual Control Support

Identifies React-based (virtual) controls and verifies stable callback references — the #1 cause of useEffect re-render loops in PCF.

10Resource Leak Detection Production guardrail

After destroy() is called, the harness reports resources that were not cleaned up — the leaks that kill Field Service Mobile performance over long sessions.

Event Listeners

Monkey-patched addEventListener tracks every registration on window, document, and the control container. Unmatched listeners flagged by type, target, and stack trace.

Timers

setInterval calls and long-lived setTimeout (> 5 s) tracked. Uncleaned timers in PCF keep controls alive across navigation — a major mobile battery drain.

Observers

MutationObserver, ResizeObserver, IntersectionObserver instances tracked. Unresolved observers hold strong references that prevent GC.

11Hot Reload

Instant feedback loop: edit your control, rebuild, and see changes with no manual browser refresh.

File Watching

Vite plugin watches for changes to bundle.js in the control's out/ directory. Survives npm run build restarts.

500 ms Debounce

Rapid file changes debounced to avoid redundant reloads during incremental builds.

Full Control Reinit

On reload the control goes through the full lifecycle: destroy() then init() + updateView(). Metrics and logs reset for a clean baseline.

13getResource & RESX Localization

All bundle resources (images, fonts, RESX) preloaded at startup so getResource() returns base64 data synchronously — matching real Dynamics 365 timing.

Preload Cache

context.resources.getResource() returns base64 data synchronously from an in-memory cache populated at startup by scanning the bundle output directory. Matches Dynamics runtime timing so async-related bugs surface locally.

RESX Parsing

.resx files referenced in the manifest are parsed into a key-value map. context.resources.getString(key) returns the localized string. Uses fast-xml-parser with graceful fallback on errors.

  • Extracts <data name="key"><value> entries
  • Supports multiple .resx files per control
  • Synchronous resource access matches real runtime
  • Handles images, fonts, and localization strings

17CSS Isolation @layer pcf-control

Control CSS is loaded into a CSS layer so Bootstrap resets and global styles from one control can't break the harness UI or sibling controls.

How it works /* Harness Fluent UI styles — unlayered (highest priority) */
.harness-topbar { background: white; }

/* Control CSS loaded into pcf-control layer (lower priority) */
@layer pcf-control {
  * { margin: 0; } /* Bootstrap reset — won't escape */
  body { font-size: 12px; } /* scoped to control */
}

Harness Always Wins

Unlayered styles always take priority over layered ones via CSS cascade rules — controls can't accidentally restyle harness chrome.

Controls Still Work

Within the control container, layered styles still apply normally. Your body resets and global selectors work; they just don't escape.

Why PCF Workbench?

The pcf-scripts start test harness was designed to answer one question: "does my control load?" Workbench was built to answer every question that comes after.

The problem it solves

Every PCF team hits the same wall. The built-in harness shows the control in isolation on a desktop browser — but controls in production run on phones, with flaky networks, inside Mobile Offline Profiles, against real Dataverse schemas.

Standard pcf-scripts start Pain

  • One control at a time — no way to browse a project of 20 controls
  • No device emulation — you ship to mobile hoping it fits
  • No network simulation — offline-first is blind until production
  • No WebAPI — every test requires pushing to a real environment
  • No lifecycle visibility — re-render loops surface as "it feels slow"
  • No memory / leak detection — Field Service Mobile dies in the field
  • No reproducible scenarios — bug repros are tribal knowledge
  • Slow feedback loop — build, push, open form, reproduce, repeat

With PCF Workbench Win

  • Gallery of every control, searchable, with thumbnails & build status
  • Four device presets + CSS container queries for true responsive testing
  • Offline / Slow 3G / Fast 3G / Custom latency, applied to every WebAPI call
  • WebAPI mock with full OData $filter, $select, $orderby, paging
  • Lifecycle monitor with health checks for re-render loops and missing cleanup
  • Resource leak detection: listeners, timers, observers reported after destroy
  • Save / load / export test scenarios — ship bug repros with the PR
  • Hot reload on bundle.js — no deploys to iterate

Who it's for

PCF Workbench is valuable at three points in the PCF lifecycle.

Author

PCF developers building new controls

Fast iteration, visibility into every lifecycle hook, property-editor generated from the manifest. Stop pushing half-built controls to a sandbox just to see them render.

Maintainer

Teams inheriting legacy controls

Resource leak detection and render timeline turn a vague "it's slow on mobile" bug into a specific line in a specific hook. Gallery mode makes a 30-control project browsable.

Reviewer

Tech leads and reviewers

Load any control, load any saved scenario, eyeball the render timeline and WebAPI log. Approve PRs with evidence, not faith.

Mobile-First Dev

Field Service Mobile engineers

Offline-mode routing, Mobile Offline Profile simulation, CSS container queries for accurate responsive layouts at phone sizes. The mobile bugs that used to need a real device now surface in dev.

What a fix actually costs

A typical iteration — reproducing and fixing a bug that only appears when the control is disabled on a phone, offline.

Without the Workbench

  1. Edit TypeScript, run npm run build
  2. pac pcf push --publisher-prefix cli
  3. Open model-driven app in browser
  4. Switch to mobile emulation in DevTools
  5. Log in, navigate to the form, open the record
  6. Bump manifest version, re-push (cache)
  7. Open dev tools, try to repro offline by disabling network
  8. Realise disabled state isn't in browser mode — ship to a real phone
  9. Hand phone to tester, hope they can repro
Typical cycle 15–20 min

With PCF Workbench

  1. Edit TypeScript, run npm run build
  2. Harness hot-reloads the bundle automatically
  3. Load saved scenario: offline-phone-disabled
  4. Check lifecycle + render timeline for the bug
  5. Fix. Reload. Confirm leak detector is clean.
Typical cycle < 60 sec
